# project_rules.md

## 1. 文档目的

本文件用于约束本项目在开发阶段的实现方式、代码风格、交付边界与决策优先级。

它服务于以下目标：

1. 保证实现结果与冻结需求保持一致
2. 防止 Agent 或开发者在 M0 阶段擅自扩展范围
3. 统一前端技术路线为 React，后端技术方向为 Python
4. 让每一轮开发都能被审阅、追踪和校验

---

当前开发任务已切换到 **M0 阶段（后端数据源接入）**。

## 3. 技术选型规则

### 3.1 前端

本项目开发阶段前端统一优先采用：

- React
- TypeScript
- Vite
- TailwindCSS
- Zustand
- MapLibre GL JS
- ECharts

说明：

- 即使需求冻结文档中出现过 Vue 3 / Pinia，也不作为当前开发阶段强制要求。
- 在不改变业务功能和口径的前提下，允许以 React 等价实现。

### 3.2 后端

M0 阶段正式引入后端，采用以下技术栈：

- Python 3.11+
- FastAPI
- Pydantic
- SQLModel
- SQLite（轻量数据底座）
- Alembic（数据库迁移）
- Uvicorn

禁止在 M0 阶段引入：

- Java / Spring Boot
- Node.js 后端
- Go 后端
- 数据库中间件
- MQ
- 缓存服务

## 4. 架构约束规则

### 4.1 前后端分离架构

M0 阶段采用前后端分离架构：

- 前端：`frontend/` 目录，React 单体应用
- 后端：`backend/` 目录，FastAPI 服务
- 默认模式：前端使用 `local` 模式读取本地 JSON
- API 模式：前端可切换从后端 API 读取数据

### 4.2 前端模块分层

前端 `frontend/` 目录应保持以下分层结构：

1. **原始数据层** - 从 JSON 或 API 读取原始数据
2. **清洗归一化数据层** - dataParser.ts 处理后的干净数据
3. **当前筛选结果层** - 根据筛选条件过滤后的数据

禁止组件直接反复处理原始 JSON 脏数据。

### 4.3 展示与计算分离

必须将以下逻辑从 UI 组件中拆分出来：

- 数据清洗
- 筛选计算
- 统计计算
- 枚举映射
- 风险颜色映射
- 电压映射

UI 组件只负责展示与交互，不负责承载复杂数据规则。

### 4.4 后端模块分层

后端 `backend/app/` 目录应保持以下分层结构：

1. **api/** - 路由定义
2. **core/** - 核心配置（数据库连接、应用配置）
3. **importers/** - 数据导入服务
4. **models/** - 数据模型（SQLModel）
5. **schemas/** - 响应模型（Pydantic）

## 5. 数据规则

### 5.1 坐标规则

以下记录必须过滤：

- 经纬度为空
- 经纬度非数字
- 经纬度超出湖南范围
- 经纬度为 0

这些记录：

- 不上图
- 不参与统计
- 不得导致页面报错

### 5.2 人数规则

#### 半径映射值

- 空值 / 非数字 / 负数 → 0
- 0~300 → 原值
- > 300 → 300

#### 统计与详情值

- 空值 / 非数字 / 负数 → 0
- 其他 → 原值

### 5.3 风险规则

- 1~4 按正常等级处理
- 空值 / 非法值 → 未知风险
- 未知风险颜色固定为灰色
- 风险统计中必须单列

### 5.4 状态规则

- 仅支持：全部 / 作业中 / 作业暂停 / 当日作业完工
- 其他值统一归类为未知状态
- 未知状态默认仅在"全部"下显示

### 5.5 市州规则

- 市州筛选采用固定业务枚举
- 枚举外值归入"其他"
- "其他"不参与 Top5

### 5.6 同坐标多记录规则

- 允许点位完全重叠
- 点击仅展示最上层记录
- 仅在命中重叠点时显示提示
- 不得提前实现聚合或散开策略

### 5.7 数据版本规则（M0 新增）

- 每次导入必须记录导入批次（UUID）
- 每次导入必须记录导入时间
- 每次导入必须记录源文件名
- 每次导入必须记录导入人
- 同主键数据必须全覆盖更新
- 导入日志必须保留，不得删除

## 6. 统计规则

所有统计必须基于**当前筛选结果中的合法坐标记录**。

### 6.1 当日作业点总数

= 合法坐标记录数（指当前选中日期）

### 6.2 当前作业总人数

= 合法坐标记录的统计/详情值之和

### 6.3 风险分布

= 按记录数统计

### 6.4 市州分布 Top5

= 按记录数统计，不含"其他"

### 6.5 一致性校验

必须满足：

`地图标记数量 == 当日作业点总数`

若不满足，视为实现错误，而不是需求歧义。

---

## 7. 交互规则

### 7.1 页面结构

页面必须保持：

- 左：筛选
- 中：地图
- 右：详情
- 底：统计

### 7.2 必要交互

- hover 显示 Tooltip
- click 打开详情
- 筛选实时联动地图与统计
- 选中项被筛掉后自动清空详情

### 7.3 页面状态

必须覆盖：

- loading
- load error
- empty data
- filtered empty

---

## 8. UI 与视觉规则

### 8.1 视觉优先级

优先级如下：

1. 信息准确
2. 联动清晰
3. 布局稳定
4. 视觉统一
5. 动效美观

### 8.2 明确禁止

禁止引入以下视觉增强：

- 3D 建模
- 粒子
- 飞线
- 地图脉冲动画
- 炫光扫描
- 大量无业务价值装饰

### 8.3 风格方向

可参考 worldmonitor 的布局组织，但不得照搬效果，不得以"科技感"为主要目标。

## 9. 代码组织规则

### 9.1 基本要求

- 使用 TypeScript
- 避免 any
- 函数职责单一
- 组件职责单一
- 不在 App 中堆积业务逻辑

### 9.2 前端模块边界

建议至少拆出：

- 地图模块
- 数据解析模块
- 筛选模块
- 统计模块
- 枚举与配置模块
- 页面状态模块
- 时间轴模块

### 9.3 后端模块边界

建议至少拆出：

- API 路由层
- 核心配置层
- 数据导入服务层
- 数据模型层
- 响应模型层

### 9.4 命名要求

- 命名清晰，避免缩写泛滥
- 与业务字段相关的命名必须可追踪
- 枚举与映射需集中管理

---

## 10. 开发流程规则

### 10.1 逐轮推进

禁止一轮生成"完整项目终稿"。

应按以下方式推进：

1. 一轮只实现一个清晰子目标
2. 每轮说明输入、输出、风险和下一步
3. 每轮结束后可被评审

### 10.2 变更控制

若出现以下情况，必须暂停并单独说明：

- 需求文档无法支撑实现
- 现有技术路线与需求冲突
- React 替代原 Vue 写法会影响需求口径
- 必须引入新的外部服务才能推进

未经说明不得自行改变架构。

---

## 11. 偏离规则

若开发过程中需要偏离冻结文档，必须满足以下条件：

1. 明确指出偏离点
2. 说明偏离原因
3. 说明影响范围
4. 说明是否影响验收
5. 先提出建议，再执行修改

禁止静默偏离。

---

## 12. 验收规则

实现必须以冻结需求文档中的验收项为准。

开发者或 Agent 不得用"实现方便"修改验收口径。

尤其必须确保：

- 地图标记数量与点位统计一致
- 风险颜色正确
- 详情字段与记录一致
- 异常数据不导致白屏
- 筛选无结果有明确提示
- M0 后端接口可正常调用
- M0 数据导入功能可正常执行
- M0 前端 API 模式切换可正常工作

---

## 13. 最终原则

本项目的 M0 阶段目标是：

- 小步可交付
- 功能闭环清晰
- 数据口径可靠
- 后端底座稳定
- 后续可扩展
- 当前不越界

任何"为了未来预埋完整架构"的冲动，都不能优先于当前冻结范围。
