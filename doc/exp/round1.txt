## 第三轮需求分析：从数字沙盘到"基建online"

### 一、愿景理解

你的构想是将输变电工程管理**游戏化**，借鉴《三国志》《文明》等4X策略游戏的核心循环：

| 游戏元素 | 工程映射 | 游戏机制 |
|---------|---------|---------|
| **军队/工人** | 作业人数 | 可投入建设的资源单位 |
| **城市** | 变电站 | 建设目标，产出"电力价值" |
| **道路** | 输电线路（杆塔序列） | 连接城市的网络，需逐塔建设 |
| **赛季** | 年度管理周期 | 排行榜、成就结算 |
| **势力** | 建管单位 | 成就值（管理效能） |
| **武将/军团** | 施工单位 | 战斗力（施工能力） |

**核心游戏循环**：
```
投入人力 → 建设杆塔 → 贯通线路 → 变电站投产 → 获得成就值
     ↑                                              ↓
     ←←←←←←←← 持续优化资源配置 ←←←←←←←←←←←←←←←←←
```

### 四、游戏化机制设计建议

#### 4.1 核心数值系统

```
建管单位成就值 = Σ(投产变电站容量 × 难度系数) + Σ(按期完工线路 ×  bonus)
                ─────────────────────────────────────────────────────
                                    总工期压缩率

施工单位战斗力 = 月均投入人时 × 安全事故系数 × 质量评分 × 进度达成率
```

#### 4.2 赛季制设计

| 赛季阶段 | 时长 | 活动内容 |
|---------|-----|---------|
| **S1 筹备期** | 1月 | 年度工程计划发布，各单位组建"军团" |
| **S2 建设期** | 10月 | 主线 gameplay，建设、竞争、协作 |
| **S3 冲刺期** | 1月 | 年末抢工，双倍成就值活动 |
| **S4 结算期** | 1月 | 排行榜公布、成就展示、下赛季预告 |

#### 4.3 地图可视化升级

```
当前：散点图（站班会位置）
  ↓
第三阶段：分层可视化
  ├── 底图：湖南省行政区划
  ├──  layer 1：变电站（城市图标，大小=容量等级）
  ├──  layer 2：输电线路（连线，颜色=建设状态）
  ├──  layer 3：杆塔（小图标，颜色=完成/在建/待建）
  └──  layer 4：实时作业点（当前已有，叠加显示）
```

### 五、技术架构演进路线

基于当前技术栈（React + Zustand + MapLibre），我建议分三期演进：

#### 5.1 第三阶段 MVP：工程骨架可视化（2-3个月）

**目标**：展示"城市-道路-军队"三层结构

```
新增模块：
├── data/
│   ├── towers/           # 杆塔坐标数据（静态JSON）
│   ├── stations/         # 变电站坐标数据
│   └── lines/            # 线路拓扑关系
├── src/
│   ├── features/
│   │   ├── network/      # 线路/杆塔图层
│   │   └── progress/     # 进度可视化组件
│   └── lib/
│       ├── networkBuilder.ts  # 构建线路-杆塔网络
│       └── progressCalculator.ts # 进度计算
```

**关键实现**：
- 使用 MapLibre 的 `LineLayer` 绘制线路
- 杆塔用 `CircleLayer`，按建设状态着色
- 变电站用自定义图标（城市样式）
- 进度数据用静态JSON模拟（等待真实数据源）

#### 5.2 第四阶段：实时态势接入（2-3个月）

**目标**：接入实时进度数据，动态更新建设状态

```
架构升级：
┌─────────────────────────────────────────┐
│           前端 (React + MapLibre)        │
├─────────────────────────────────────────┤
│  WebSocket Client  ←→  态势编排层        │
├─────────────────────────────────────────┤
│      Python FastAPI (数据接入网关)       │
├─────────────────────────────────────────┤
│   时序库(TimescaleDB)  ←→  外部数据源    │
└─────────────────────────────────────────┘
```

#### 5.3 第五阶段：游戏化系统（2-3个月）

**目标**：成就系统、赛季、排行榜完整闭环

```
新增系统：
├── 成就系统（Achievement System）
│   ├── 建管单位成就值计算
│   ├── 施工单位战斗力计算
│   └── 徽章/称号体系
├── 赛季系统（Season System）
│   ├── 赛季时间轴
│   ├── 排行榜（实时/赛季末）
│   └── 历史赛季回放
└── 社交系统（可选）
    ├── 单位间协作关系图
    └── 工程经验分享
```

### 六、风险与建议

#### 6.1 主要风险

| 风险 | 影响 | 缓解措施 |
|-----|-----|---------|
| **数据源延迟** | 杆塔/进度数据无法及时到位 | 先用mock数据跑通流程，数据到位后无缝切换 |
| **性能瓶颈** | 杆塔数量可能达10万级 | 引入deck.gl处理海量数据，或按层级聚合 |
| **游戏化设计** | 业务方不接受"游戏"定位 | 定位为"工程态势 gamification"，强调管理价值 |
| **复杂度失控** | 功能蔓延导致无法交付 | 严格分阶段，每阶段可独立演示 |

#### 6.2 我的建议

**短期（立即启动）**：
1. **确认数据接口**：与提供杆塔/进度数据的团队对齐时间表
2. **设计数值公式**：先定义成就值和战斗力的计算公式
3. **制作原型**：用Figma或代码快速做一个"游戏化"风格的原型，验证视觉方向

**中期（第三阶段）**：
1. 先实现**静态的工程骨架可视化**（杆塔+线路+变电站）
2. 使用**模拟进度数据**演示建设过程动画
3. 设计并演示**赛季时间轴**（可切换历史赛季）

**长期（第四、五阶段）**：
1. 接入真实进度数据源
2. 实现完整的游戏化系统
