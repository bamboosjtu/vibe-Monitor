# 输变电工程数字沙盘系统 - 数据维护说明

**版本**: M1 Round2 当前实现  
**日期**: 2026-05-03

---

## 一、数据文件命名规范

### 1.1 标准格式

```
meeting_YYYYMMDD_all.json
```

**说明**:
- `meeting_`: 固定前缀
- `YYYYMMDD`: 8位日期数字（如 20260401）
- `_all`: 固定后缀，表示全天数据
- `.json`: 文件扩展名

**示例**:
- `meeting_20260401_all.json` - 2026年4月1日数据
- `meeting_20260402_all.json` - 2026年4月2日数据

### 1.2 非法命名示例

| 文件名 | 是否合法 | 原因 |
|--------|----------|------|
| `20260401.json` | ❌ | 缺少前缀和后缀 |
| `meeting_20260401.json` | ❌ | 缺少 `_all` 后缀 |
| `meeting_2026-04-01_all.json` | ❌ | 日期格式错误（含连字符）|
| `meeting_20260401_all.txt` | ❌ | 扩展名错误 |

## 二、数据目录结构

### 2.1 目录作用

```
项目根目录/
├── data/                          # 数据文件目录（数据源）
│   ├── daily_meeting/
│   │   ├── meeting_20260401_all.json
│   │   ├── meeting_20260402_all.json
│   │   └── ...
│   ├── tower/
│   ├── year_progress_formation/
│   ├── substation_coordinates/
│   └── schema/
│
├── frontend/public/data/          # 静态资源目录（供前端访问）
│   └── daily_meeting/index.json   # 日期清单（manifest）
│
└── frontend/scripts/
    └── generate-data-index.js     # 自动生成 manifest 脚本
```

### 2.2 目录说明

| 目录 | 作用 | 内容 | 系统如何读取 |
|------|------|------|-------------|
| `data/daily_meeting/` | **站班会数据源目录** | 存放所有日期数据文件 `meeting_YYYYMMDD_all.json` | 开发/预览由 Vite 数据插件读取，生产需部署到 Web `/data/daily_meeting/` |
| `frontend/public/data/` | **静态资源目录** | 存放 `daily_meeting/index.json`（日期清单） | 系统**通过此目录的 index.json 发现可用日期** |

**重要说明**：
- 系统将 `frontend/public/data/daily_meeting/index.json` 作为识别可用日期的**唯一入口**
- `data/daily_meeting/` 存放每日站班会源文件，需要通过脚本生成 `index.json` 后才能被时间轴识别
- 前端请求路径固定为 `/data/daily_meeting/*`；开发/预览环境由 `frontend/vite.config.ts` 映射到仓库根目录 `data/`
- 新增数据文件后，必须运行 `npm run generate:data-index` 更新 `index.json`

## 三、日期清单（index.json）

### 3.1 文件作用

`frontend/public/data/daily_meeting/index.json` 是系统识别可用日期的**唯一入口**：

- **系统启动时读取此文件** - 从 `/data/daily_meeting/index.json` 路径加载
- **根据 `dates` 数组生成时间轴范围** - 决定"上一天"/"下一天"的导航边界
- **默认加载 `dates` 数组中的最后一个日期** - 最新日期作为初始日期
- **动态发现机制** - 无需修改代码，只需更新此文件即可扩展日期范围

### 3.2 文件格式

```json
{
  "dates": [
    "2026-04-01",
    "2026-04-02",
    "2026-04-03",
    "2026-04-04",
    "2026-04-05",
    "2026-04-06",
    "2026-04-07",
    "2026-04-08",
    "2026-04-09",
    "2026-04-10",
    "2026-04-11"
  ],
  "meta": {
    "description": "输变电工程数字沙盘系统 - 可用数据日期清单",
    "format": "YYYY-MM-DD",
    "lastUpdated": "2026-04-12"
  }
}
```

**字段说明**:
- `dates`: 日期数组，格式为 `YYYY-MM-DD`，按升序排列
- `meta.description`: 描述信息
- `meta.format`: 日期格式说明
- `meta.lastUpdated`: 最后更新时间

### 3.3 注意事项

- `dates` 中的每个日期必须在 `data/` 目录中有对应的文件
- 日期格式必须严格为 `YYYY-MM-DD`（如 `2026-04-01`）
- 系统会自动按升序排序，无需手动排序

## 四、自动生成脚本

### 4.1 脚本作用

`frontend/scripts/generate-data-index.js` 自动扫描仓库根目录的 `data/daily_meeting/` 目录，生成 `frontend/public/data/daily_meeting/index.json`：

- 识别所有符合 `meeting_YYYYMMDD_all.json` 格式的文件
- 提取日期并自动排序
- 生成标准的 `index.json` 文件

**脚本位置**：`./frontend/scripts/generate-data-index.js`

### 4.2 使用方法

```bash
# 运行脚本
cd frontend
npm run generate:data-index
```

**输出示例**:
```
[Generate Data Index] 开始生成日期清单...
[Generate Data Index] 扫描目录: C:\...\data\daily_meeting
[Generate Data Index] 发现 11 个日期文件
[Generate Data Index] 日期列表: 2026-04-01, 2026-04-02, ..., 2026-04-11
[Generate Data Index] 成功生成: C:\...\frontend\public\data\daily_meeting\index.json
[Generate Data Index] 时间范围: 2026-04-01 至 2026-04-11
[Generate Data Index] 完成!
```

### 4.3 何时运行

| 场景 | 操作 |
|------|------|
| 新增日期数据文件 | 运行脚本更新 `index.json` |
| 删除日期数据文件 | 运行脚本更新 `index.json` |
| 重命名数据文件 | 运行脚本更新 `index.json` |
| 首次部署 | 运行脚本生成初始 `index.json` |

## 五、新增一天数据的标准步骤

### 5.1 准备数据文件

1. 获取新日期的数据（JSON 格式）
2. 确保数据包含 `list` 数组

### 5.2 命名文件

按规范命名文件：

```
meeting_YYYYMMDD_all.json
```

**示例**: 2026年4月12日的数据文件命名为 `meeting_20260412_all.json`

### 5.3 放入目录

将文件复制到 `data/daily_meeting/` 目录：

```bash
cp meeting_20260412_all.json data/daily_meeting/
```

### 5.4 更新清单

运行自动生成脚本：

```bash
cd frontend
npm run generate:data-index
```

脚本会自动：
- 扫描 `data/daily_meeting/` 目录
- 识别新文件
- 更新 `frontend/public/data/daily_meeting/index.json`

### 5.5 验证

1. 检查脚本输出，确认识别到新日期
2. 打开 `frontend/public/data/daily_meeting/index.json`，确认新日期已添加
3. 刷新页面，确认时间轴范围已扩展

## 七、常见问题

### 7.1 时间轴不显示新日期

**原因**: `index.json` 未更新

**解决**:
```bash
cd frontend
npm run generate:data-index
```

### 7.2 脚本报错"数据目录不存在"

**原因**: 运行目录不正确

**解决**: 在 `frontend/` 目录运行 `npm run generate:data-index`

### 7.3 文件命名正确但未被识别

**排查**:
1. 检查文件名是否严格符合 `meeting_YYYYMMDD_all.json`
2. 检查文件是否在 `data/daily_meeting/` 目录下
3. 检查文件扩展名是否为 `.json`

### 7.4 时间轴显示日期但数据加载失败

**原因**: `index.json` 中的日期与实际文件不匹配

**解决**:
1. 检查 `data/daily_meeting/` 目录是否存在对应文件
2. 重新运行 `npm run generate:data-index`

## 八、数据文件格式要求

### 8.1 基本结构

```json
{
  "code": "200",
  "data": {
    "list": [
      {
        "id": "记录ID",
        "prjName": "工程名称",
        "toolBoxTalkLongitude": "112.5",
        "toolBoxTalkLatitude": "28.3",
        "currentConstrHeadcount": 10,
        "reAssessmentRiskLevel": 2,
        ...
      }
    ]
  }
}
```

### 8.2 关键字段

| 字段 | 说明 | 是否必需 |
|------|------|----------|
| `id` | 记录唯一标识 | 是 |
| `prjName` | 工程名称 | 是 |
| `toolBoxTalkLongitude` | 经度 | 是（用于地图定位）|
| `toolBoxTalkLatitude` | 纬度 | 是（用于地图定位）|
| `currentConstrHeadcount` | 当前作业人数 | 否（为空时视为0）|
| `reAssessmentRiskLevel` | 风险等级 | 否（为空时视为unknown）|

## 九、维护检查清单

| 检查项 | 频率 | 操作 |
|--------|------|------|

## 十、M0 后端数据导入（M0 新增）

M0 阶段新增了后端数据导入能力，支持通过 API 导入数据到 SQLite 数据库。

### 10.1 后端数据库初始化

```bash
cd backend
uv run python scripts/db/init_db.py
```

这将创建或确认 `backend/data/sandbox.db` 和相关表结构。

### 10.2 数据导入方式

M0 阶段支持两种数据导入方式：

#### 方式一：本地静态 JSON（默认）

沿用原有方式，将 JSON 文件放入 `data/daily_meeting/` 目录，运行 `npm run generate:data-index`。

#### 方式二：后端 API 导入

通过后端 API 导入数据到数据库：

| 接口 | 方法 | 说明 |
|------|------|------|
| `POST /api/import/year-progress` | 导入年度目标数据 | 支持全覆盖更新 |
| `POST /api/import/towers` | 导入杆塔数据 | 支持全覆盖更新 |
| `POST /api/import/meetings` | 导入 meetlist 数据 | 写入最新快照 |

### 10.3 数据版本留痕

每次导入都会在数据库中记录：
- 导入批次号（UUID）
- 导入时间
- 导入数据源类型
- 源文件名
- 导入记录数
- 导入人

### 10.4 数据源切换

前端默认使用 `local` 模式读取本地 JSON。如需切换到 API 模式：

```javascript
import { useAppStore } from '@/store';
useAppStore.getState().setDataSource('api');
```

> 注：API 模式下需要后端服务已启动（`uv run python run.py`）。
