# 一、《基建online》PRD（产品需求文档）

## 1. 产品概述

### 1.1 产品名称

**基建online：输变电工程调度指挥沙盘**

### 1.2 产品定位

基建online是面向输变电工程建设管理场景的**严肃游戏化调度指挥沙盘**。
它不是独立的新业务体系，而是在现有工程管理、数字工单、项目组织、现场作业与评价流程之上，构建一套更易理解、易参与、易统筹、易复盘的可视化与指挥化系统。

其建设方向应服务于公司既有的数智化转型目标：以数字工单为基础，形成“业务线上流转 + 现场智能感知 + 远程集中监控”的新型基建管理模式，并支撑项目全过程调度指挥、预警督办和过程评价。

### 1.3 产品价值

产品核心价值分为五类：

1. **全局态势可见**：从“看点位”升级为“看组织、看项目、看事件、看工单、看健康度”。
2. **调度指挥可执行**：把军令映射为真实责任动作，落入工单和验收链。
3. **过程闭环可追踪**：从事件触发、工单下发、现场执行、验收复核到结算归档全链条留痕。
4. **绩效评价可到人**：既评价单位，也评价军师、主将等关键岗位。
5. **能力建设可沉淀**：通过履历系统形成长期的人才、项目、组织资产。

### 1.4 建设背景

现代建设管理体系已明确提出，要抓实“前期、业主、监理、施工”四大项目部，分别承担保障单元、指挥单元、指挥支撑单元和作战单元职责，并提升业主监理项目部指挥能力、突出施工项目部作战能力，贯通全过程进度管理。

基建数智化管控中心工作手册进一步明确，管控中心是建设管理部门的“神经中枢”和“参谋中心”，以数字工单为抓手，构建“过程监控、分析预警、问题督办、考核评价”的监督机制，并形成省级统筹预警、市州执行整改的分层管控体系。

在此基础上，基建online承担的是“业务机制表达层 + 可视化指挥层 + 过程闭环层”的角色，而不是替代既有业务系统。

## 2. 产品目标

## 2.1 总目标

建设一个可支撑省、市、项目三级协同的输变电工程调度指挥沙盘，使其同时具备：

- 态势感知能力
- 项目指挥能力
- 事件预警与督办能力
- 单位与个人双线评价能力
- 关键岗位履历沉淀能力
- 分阶段、分机制包独立上线能力

## 2.2 分目标

### 目标A：看清全局

支持GM、战区、项目三个层级快速掌握：

- 项目分布
- 当前作业
- 风险热区
- 工单积压
- 军团承载
- 健康度异常

### 目标B：形成指挥闭环

所有军令必须具备：

- 发起对象
- 承接对象
- 截止时间
- 证据要求
- 验收链
- 战绩影响

### 目标C：评价到人

形成三类账本：

- 单位战绩
- 项目战果
- 个人战绩

### 目标D：履历留痕

对军师、主将及关键岗位：

- 记录任- 记录任命
- 记录调整
- 记录接任与调离
- 区分完整履历与阶段履历

### 目标E：模块化上线

将系统拆为可独立建设、独立灰度、独立上线的机制包，降低研发耦合度。

## 3. 用户与角色模型

基建online采用四层角色模型：

**GM总控层 — 战区指挥层 — 现场督军层 — 军团作战层**

这一拆分既符合现代建设管理体系中四大项目部与分层调度的职责划分，也符合两级管控中心的iile0

## 3.1 GM总控层

**业务对应**：省建设部 / 省级管控中心
**职责**：

- 发布赛季目标
- 发布规则版本
- 监控全省态势
- 发起提级督办
- 裁决重大申诉
- 发布赛季复盘

## 3.2 战区指挥层

**业务对应**：建管单位 / 督军府 / 军师体系

内部拆为三层：

- **战区 / 势力** = 建管单位
- **督军府** = 业主项目部 / 建管侧项目指挥单元
- **军师** = 业主项目经理个人席位

说明：

- 一个督军府可包含多个军师
- 军师是个人战绩与履历的核心对象
- 战区负责目标承接和区域统筹
- 督军府负责项目组织与协同

## 3.3 现场督军层

**业务对应**：监理项目部
**职责**：

- 巡视、旁站
- 转序验收
- 问题整改发起
- 复工核验
- 日评价支撑
- 对作战军形成现场约束

监理项目部在现代建设管理体系中属于指挥单元的重要组成，协助业主项目部做好现场“四控二管ile1

## 3.4 军团作战层

**业务对应**：施工单位 / 施工项目部 / 作业班组

内部拆为四层：

- **军团** = 施工单位
- **作战军** = 施工项目部
- **主将** = 施工项目经理
- **副将 / 士兵** = 总工、安全员、技术员、班组作业人员

说明：

- 建管单位与施工单位是多对多关系
- 军团负责组建作战军
- 作战军是现场执行单元
- 主将是直接责任人

现代建设管理体系已明确提出，施工项目部是一线“作战单元”，项目经理是核ile1

## 6. 详细需求

## 6.1 M1：赛季与目标机制

### 功能目标

把年度管理要求转成系统中可持续运营的赛季体系。

### 需求点

1. 支持创建赛季
2. 支持发布赛季目标
3. 支持按战区拆解目标
4. 支持显示开工/投产完成率
5. 支持赛季榜单
6. 支持赛季归档

### 输出

- 赛季总览页
- 战区目标页
- 项目目标页
- 榜单页

## 6.2 M2：角色与席位机制

### 功能目标

将组织关系与责任席位标准化。

### 需求点

1. 支持维护战区、督军府、军师、军团、作战军主数据
2. 支持一个督军府配置多个军师
3. 支持一个项目绑定多个军师
4. 支持一个项目绑定监理项目部
5. 支持一个项目绑定多个军团
6. 支持作战军与主将绑定
7. 支持关键岗位变更记录

### 输出

- 角色关系图
- 项目组织页
- 人员席位配置页

------

## 6.3 M3：事件中心机制

### 功能目标

让系统从“静态展示”变成“动态关注重点”。

### 需求点

1. 支持事件生成
2. 支持事件列表与筛选
3. 支持事件详情
4. 支持事件状态流转
5. 支持事件与工单关联
6. 支持热点事件聚合
7. 支持人工确认和清除

### 输出

- 事件中心页
- 热点列表
- 事件详情抽屉

------

## 6.4 M4：军令与工单机制

### 功能目标

把所有前台调度动作落为后台责任动作。

### 军令类型

- 督战令
- 支援令
- 调整令
- 请示令
- 整改令
- 复工令

### 工单状态

- 待接收
- 处理中
- 待验收
- 已闭环
- 已驳回
- 已超期
- 已冻结

### 需求点

1. 支持创建军令
2. 支持工单接收与流转
3. 支持SLA管理
4. 支持上传证据
5. 支持验收通过 / 驳回
6. 支持工单超期自动标记
7. 支持工单轨迹展示

### 输出

- 工单中心
- 工单详情
- 我的工单
- 超期工单看板

------

## 6.5 M5：现场日循环机制

### 功能目标

形成线上线下联动的最小闭环。

### 微观链路

作业票 → 站班会 → 当日作业 → 日评价 → 问题整改 → 闭环

这与现有工单体系中的“现场每日作业”“监理旁站”“监理巡视”“检查问题通知单”“日评价”ile0

### 需求点

1. 支持显示当日作业
2. 支持接入站班会状态
3. 支持监理旁站与巡视记录
4. 支持创建问题整改工单
5. 支持军师/监理对作战军日评价
6. 支持日循环结果写入项目态势

### 输出

- 日循环页
- 当日作业清单
- 日评价入口
- 整改单入口

------

## 6.6 M6：健康度机制

### 功能目标

用一个综合状态表达项目当前运行质量。

### 维度

- 进度
- 安全
- 质量
- 资源
- 协同

### 输出

- 健康分
- 健康等级（绿/黄/红/灰）
- 异常原因标签
- 趋势曲线

### 需求点

1. 支持自动计算健康分
2. 支持按项目显示健康等级
3. 支持按战区聚合健康分布
4. 支持因事件与工单变化刷新健康度

------

## 10. 工单状态机

```text
新建 → 待接收 → 处理中 → 待验收
                       ├→ 已闭环
                       ├→ 已驳回 → 处理中
                       ├→ 已超期 → 处理中/待验收
                       └→ 已冻结 → 待裁决
```

### 状态规则

1. 待接收必须记录工单创建来源。
2. 处理中必须记录接单人和接单时间。
3. 待验收必须包含最小证据包。
4. 已驳回必须写入驳回原因。
5. 已超期自动生成超期事件。
6. 已冻结暂停加分发放，但保留状态显示。

## 11. 战绩公式

## 11.1 通用公式

```
总战绩 = 节点分 + 过程分 + 协同分 + 荣誉分 - 扣分项 ± 调整项
```

## 11.2 军师个人战绩

```
军师战绩 = 0.35*节点达成分 + 0.25*协调闭环分 + 0.15*资源匹配分 + 0.15*日评价加权分 + 0.10*专项荣誉分 - 违规/超期扣分
```

## 11.3 主将个人战绩

```
主将战绩 = 0.30*进度执行分 + 0.25*质量分 + 0.25*安全分 + 0.10*资源利用分 + 0.10*协同响应分 - 违规/停工/返工扣分
```

## 11.4 现场督军战绩

```
现场督军战绩 = 0.30*巡视旁站覆盖分 + 0.25*问题发现有效分 + 0.20*整改闭环分 + 0.15*验收准确率 + 0.10*日评价有效性 - 漏检/误判扣分
```

## 11.5 军团单位战绩

```
军团战绩 = 0.25*主将平均分 + 0.20*项目履约率 + 0.20*安全质量综合分 + 0.15*资源支援成效 + 0.10*战区协同评价 + 0.10*荣誉加分 - 单位级处罚
```

## 11.6 战区单位战绩

```
战区战绩 = 0.35*目标完成率 + 0.20*重点项目健康稳定度 + 0.15*协调效率 + 0.10*工单闭环率 + 0.10*军师平均表现 + 0.10*专项荣誉 - 严重异常扣分
```

## 11.7 项目战果

```
项目战果 = 0.40*节点达成 + 0.20*安全表现 + 0.20*质量表现 + 0.10*资源匹配 + 0.10*协同效率 - 停工/返工/超期扣分
```

## 11.8 结算规则

- 扣分实时生效
- 加分进入待发放池
- 争议分进入冻结池
- 裁决后转有效分或废弃分
- 年底赛季结算统一归集

## 13. 技术设计（Technical Design）

## 13.1 设计目标

技术设计目标有四项：

1. **低耦合**：按机制包拆分，不做大一统单体业务逻辑。
2. **可配置**：规则、权重、事件枚举、工单状态、战绩参数支持后台配置。
3. **可追溯**：事件、工单、结算、履历具备审计链。
4. **可演进**：前期支持静态或准实时数据，后期平滑升级到实时事件流和感知接入。

建设专业数字化顶层设计已经提出，应采用“统推自建并行、中台 + 微应用”的建设思路，基建数智化系统应延长细化e基建2.0管理链条、支撑调度指挥和过程监控；fileciteturn8file2

## 13.8 履历引擎实现建议

履历采用“记录不可改写、仅可追加”的方式。
调整动作不应直接修改历史记录，而应：

1. 关闭当前任期
2. 新建下一段任期
3. 写入交接关系

这样可避免后续任职链混乱。

## 13.10 风险与应对

### 风险一：业务口径不统一

应对：先固定角色口径、履历口径、结算口径，再开发。

### 风险二：评分引发争议

应对：先做冻结池和人工复核，再逐步自动化。

### 风险三：事件过多导致噪音

应对：事件分级 + 聚合展示 + 热点筛选。

### 风险四：页面过重

应对：P1优先做“信息架构正确”，不过度追求复杂特效。

### 风险五：与既有系统割裂

应对：坚持“表达层 + 闭环层”定位，不重造业务主流程。
