# 《基建online：输变电工程调度指挥沙盘》最终稿

**版本**：v2.0
**日期**：2026-04-14
**状态**：正式方案稿

------

## 一、项目定义

### 1.1 项目定位

基建online不是传统娱乐游戏，也不是对现有数字沙盘的简单美术升级，而是一个面向输变电工程建设管理场景的**严肃游戏化调度指挥沙盘**。

它的本质是：
以真实组织、真实流程、真实数据、真实责任为底座，以“赛季、战区、督军府、军团、军令、战绩、履历”等游戏化表达为界面语言，构建一套能够支撑**态势感知、过程指挥、异常督办、责任评价、赛季复盘**的线上线下联动系统。

该定位与公司现行数智化转型方向一致：建设专业数字化转型的总体目标，是以数字工单为基础，形成“业务线上流转 + 现场智能感知 + 远程集中监控”的基建管理新模式，并强化项目全过程调度指挥能力。

### 1.2 建设目标

基建online建设目标分为五项：

1. **让全局可见**：形成跨项目、跨区域、跨单位的一张调度态势图。
2. **让指挥可执行**：所有“军令”必须能落到真实责任对象与真实闭环动作。
3. **让过程可追踪**：从事件触发、工单下发、现场执行到验收结算形成闭环链条。
4. **让评价可到人**：既评价单位，也评价人，尤其是军师、主将及关键岗位。
5. **让系统可演进**：方案必须拆解为若干相对独立机制，以便按优先级灵活开发、独立上线、局部调整。

### 1.3 当前基础

现有数字沙盘MVP第二轮已经具备站班会点位上图、按天时间轴与历史回溯、筛选、统计和详情联动能力，这是基建online建设的良好基础。

同时，当前仍有明确边界：尚无后端服务、无WebSocket实时推送、无线路轨迹、无持久化数据库，说明下一阶段必须优先建设“机制层”和“闭环层”，而不是直接追求全量实时化与复杂三维表达。

## 二、总体设计原则

### 2.1 真实性原则

所有机制必须映射到真实业务逻辑，不能为了游戏感而制造虚构资源、虚构责任或虚构流程。

### 2.2 模块化原则

方案按独立机制包拆解，每个机制包都应满足：

- 可单独设计
- 可单独开发
- 可单独灰度
- 可单独上线
- 可单独停用或调整参数
- 不依赖全部机制同时完成才能产出价值

### 2.3 工单化原则

所有核心指挥动作必须落到“事件—工单—闭环—结算”的链条上。
公司现行管控体系本身就是以数字工单为抓手，通过计划节点自动生成工单，并依托规则预警、闭环审核与评价考核推进全过程管理。

### 2.4 到人原则

系统不能只评价项目和单位，必须支持**个人战绩、个人履历、个人调整记录**。
这一点与建设专业数字化顶层设计中提出的“强化队伍智能综合评价”“大力开展业主项目经理、施工项目经理和班组长培优行动”一致。

### 2.5 渐进演进原则

优先建设“管理闭环能力”，再建设“工程骨架能力”，最后建设“实时感知与智能联动能力”。

------

## 三、总体架构

基建online采用四层架构：

**GM总控层 — 战区指挥层 — 现场督军层 — 军团作战层**

这四层架构与现代建设管理体系的组织逻辑一致。公司已明确“前期、业主、监理、施工”四大项目部定位，其中业主项目部和监理项目部属于指挥单元，施工项目部属于作战单元；同时，施工单位贯彻“主建”、项目部“主战”的思路，施工项目部是项目建设的独立基础作战单元。

### 3.1 GM总控层

**对应业务对象**：省建设部 / 省级管控中心
**系统定位**：赛季发布者、规则制定者、全局裁判、提级介入者

### 3.2 战区指挥层

**对应业务对象**：建管单位 / 督军府 / 军师体系
**系统定位**：区域与项目指挥中枢

这一层内部拆成三层：

- **战区 / 势力** = 建管单位
- **督军府** = 业主项目部 / 建管侧项目指挥单元
- **军师** = 业主项目经理个人席位

一个督军府可包含多个军师。
军师是个人战绩与个人履历的核心载体。

### 3.3 现场督军层

**对应业务对象**：监理项目部
**系统定位**：现场督军 / 监军系统

监理不再被表达为“观察者”，而是项目现场的过程约束、验收、整改和复核力量。
这与制度中“监理项目部协助业主项目部，做好现场四控二管一协调”的定位一致。

### 3.4 军团作战层

**对应业务对象**：施工单位 / 施工项目部 / 作业队伍
**系统定位**：军团 / 作战军 / 主将副将 / 士兵

这一层内部拆成四层：

- **军团** = 施工单位
- **作战军** = 施工项目部
- **主将** = 施工项目经理
- **副将 / 士兵** = 项目关键岗位 / 班组作业人员

## 四、组织与角色体系

## 4.1 GM总控层

### 4.1.1 角色定义

GM总控层代表全省建设治理能力，不直接承担单项目作战，但控制赛季规则与重大介入权。

### 4.1.2 主要职责

- 发布赛季目标
- 发布预警规则、结算规则、履历规则
- 监控全省项目整体态势
- 对重大异常项目发起提级介入
- 裁决重大争议与申诉
- 输出赛季复盘和专题通报

------

## 4.2 战区指挥层

## 4.2.1 战区 / 势力

### 定义

战区是建管单位在系统中的区域级席位，是赛季目标分解、资源统筹、区域排名和战区复盘的主体。

### 职责

- 承接GM下达的赛季目标
- 分解到督军府、军师和项目
- 统筹战区资源
- 管理所属项目组合
- 对区域结果负责

------

## 4.2.2 督军府

### 定义

督军府是建管侧项目指挥单元，对应业主项目部。
它不是单人角色，而是一个组织席位。

### 职责

- 统筹项目节奏、计划、资源与协调
- 管理军师席位与责任分工
- 管理与军团、监理、属地、设计、物资等协同
- 形成对项目的综合评价和复盘

------

## 4.2.3 军师

### 定义

军师是督军府中的个人席位，对应一个业主项目经理。
军师是“到人评价”的核心对象之一。

### 职责

- 项目策划
- 资源协调
- 进度督导
- 问题请示
- 军令发起
- 战果承担
- 履历沉淀

### 军师特点

- 一个项目可存在多个军师
- 军师之间可区分主军师、协同军师、阶段军师
- 军师战绩按个人独立结算
- 军师调整必须记录履历，不因调岗而丢失历史贡献

------

## 4.3 现场督军层

## 4.3.1 现场督军

### 定义

现场督军对应监理项目部，是现场控制链中的关键组织。

### 职责

- 旁站
- 巡视
- 转序验收
- 问题整改发起
- 复工验证
- 日评价支撑
- 对现场履约风险提出约束意见

### 表达原则

监理既不是单纯观察者，也不是最终责任主体，而是业主项目部现场控制能力的延伸。

------

## 4.4 军团作战层

## 4.4.1 军团

### 定义

军团对应施工单位。
它是资源池、作战供给方、任命来源方，不等于具体项目上的一支部队。

### 职责

- 组建作战军
- 任命主将及关键岗位
- 统筹施工资源
- 对执行能力负责
- 对项目部履约结果承担单位责任

### 关系原则

建管单位与施工单位是**多对多关系**。
一个建管单位可与多个军团协作，一个军团也可承接多个建管单位项目。

------

## 4.4.2 作战军

### 定义

作战军对应施工项目部。
它是项目现场的直接执行单元。

### 职责

- 落实作业组织
- 管理现场人员与班组
- 完成节点施工
- 执行军令与工单
- 提交证据与反馈
- 承担安全、质量、进度、成本执行责任

------

## 4.4.3 主将与副将

### 主将

施工项目经理，是作战军核心，是项目安全、质量、进度、成本管理的直接责任人。

### 副将

副经理、总工、安全员、技术员等关键岗位，承担专业作战职责，既可独立结算战绩，也可在履历中体现任职变化。

------

## 五、模块化机制总览

为保证灵活性，基建online拆为十个相对独立的机制包。
每个机制包都可单独开发、独立上线。

| 机制包 | 名称                   | 是否可独立上线 | 主要依赖                     |
| ------ | ---------------------- | -------------- | ---------------------------- |
| M1     | 赛季与目标机制         | 是             | 基础组织数据                 |
| M2     | 角色与席位机制         | 是             | 人员、组织主数据             |
| M3     | 事件中心机制           | 是             | 基础业务状态数据             |
| M4     | 军令与工单机制         | 是             | 事件中心 / 角色权限          |
| M5     | 现场日循环机制         | 是             | 作业票 / 站班会 / 日评价     |
| M6     | 项目健康度机制         | 是             | 事件、工单、过程数据         |
| M7     | 战绩与结算机制         | 是             | 事件、工单、健康度、规则参数 |
| M8     | 履历与任免机制         | 是             | 人员、项目、组织关系         |
| M9     | 地图与态势表达机制     | 是             | 项目空间数据                 |
| M10    | 规则引擎与申诉复核机制 | 是             | 全局参数、事件与结算结果     |

## 六、机制包设计

## M1：赛季与目标机制

### 6.1 目标

把年度管理要求转成系统内持续运转的“赛季规则”。

### 6.2 核心对象

- 赛季
- 战区目标
- 督军府目标
- 军师目标
- 项目目标

### 6.3 核心规则

- 赛季默认按自然年
- 开工目标、投产目标、过程目标由GM发布
- 战区承接赛季目标并继续向项目和军师分解
- 赛季末形成单位榜、个人榜、项目榜

### 6.4 独立上线价值

即使没有完整工单闭环，赛季与目标机制也可以先上线，用于建立年度目标视图、排名视图和赛季面板。
你原策划中已有赛季结构、开工目标、投产目标和年度结算思路，这一部分可以直接演进为独立机制。

------

## M2：角色与席位机制

### 6.5 目标

把复杂组织关系固化为系统内的可计算席位体系。

### 6.6 核心对象

- GM
- 战区
- 督军府
- 军师
- 现场督军
- 军团
- 作战军
- 主将 / 副将 / 士兵

### 6.7 核心规则

- 督军府可包含多个军师
- 军团是施工单位，作战军是施工项目部
- 建管单位与军团是多对多关系
- 同一人可拥有多段履历，但同一时点不能在同一项目中占用冲突席位
- 席位变更自动触发履历写入

### 6.8 独立上线价值

这一机制可先于复杂业务流程上线，用于解决项目“谁负责、谁协同、谁执行”的问题，是后续所有战绩和工单的身份基础。

------

## M3：事件中心机制

### 6.9 目标

把分散状态变化统一收拢为“事件”。

### 6.10 事件来源

- 进度偏差
- 风险聚集
- 停工/复工
- 质量返工
- 工单超期
- 人员调整
- 节点达成
- 资源短缺
- 监理异常发现

### 6.11 核心规则

- 事件是信号，不是责任结果
- 事件可以升级、降级、清除
- 事件可触发工单
- 同类事件可聚合成热点态势

### 6.12 独立上线价值

事件中心可以先把态势看板从“静态点位”升级为“动态重点问题面板”，即使军令和结算尚未完整，也能显著提升调度价值。

------

## M4：军令与工单机制

### 6.13 目标

把“指挥动作”标准化。

### 6.14 军令类型

- 督战令
- 支援令
- 调整令
- 请示令
- 整改令
- 复工令

### 6.15 映射原则

前台叫军令，后台一定落工单。
工单必须具备：

- 发起人
- 承接人
- 截止时间
- 证据要求
- 验收链
- 状态机
- 是否影响战绩

### 6.16 状态机

建议统一工单状态：

```
待接收 → 处理中 → 待验收 → 已闭环 / 已驳回 / 已超期 / 已冻结
```

### 6.17 独立上线价值

军令与工单机制是基建online从“会看图”变成“会管事”的关键，可独立作为调度指挥模块上线。

------

## M5：现场日循环机制

### 6.18 目标

把线上线下联动的最小闭环做实。

### 6.19 日循环定义

以“作业票—站班会—现场执行—日评价—问题整改”为一个完整日循环。

这一逻辑与现有工单化实践一致：作业票是现场作业的指令单，站班会是工单执行启动会，日评价则形成逐级评价闭环。

### 6.20 核心对象

- 作业票
- 站班会
- 每日作业工单
- 日评价
- 监理旁站 / 巡视
- 问题整改工单

### 6.21 核心规则

- 有作业票才有作业事件
- 有站班会才进入有效执行状态
- 监理可发起整改单
- 军师 / 监理对作战军进行日评价
- 日循环结果进入健康度和战绩计算

### 6.22 独立上线价值

这一机制可以先在少量项目上试点上线，是最容易形成线上线下闭环价值的机制包。

------

## M6：项目健康度机制

### 6.23 目标

把项目当前运行质量表达为一个可理解、可联动、可告警的状态值。

### 6.24 健康度维度

- 进度
- 安全
- 质量
- 资源
- 协同

### 6.25 状态表达

- 绿色：正常
- 黄色：预警
- 红色：严重预警
- 灰色：停工 / 冻结

### 6.26 关键原则

- 健康度是过程态，不等于绩效分
- 健康度变化可以触发事件与工单
- 健康度仅表达当前态势，不直接作为最终战绩

### 6.27 独立上线价值

健康度机制可独立接入现有地图和详情页，让当前MVP从“只有数据”升级到“有判断”。

------

## M7：战绩与结算机制

### 6.28 目标

形成可到人、可到单位、可到项目的三类战绩账本。

### 6.29 战绩分层

- 个人战绩
- 单位战绩
- 项目战果

### 6.30 结算原则

- 扣分实时生效
- 加分延迟发放
- 重大加分可冻结
- 申诉期间进入冻结池
- 年底统一赛季结算

你原方案中关于“延迟加分、实时扣分”的思路是合理的，也适合工程管理场景，可保留并升级。

### 6.31 结算来源

- 节点达成
- 质量表现
- 安全表现
- 协同效率
- 响应时效
- 问题整改
- 资源调度成效
- 重大失误与违章

### 6.32 独立上线价值

结算模块可在工单模块之后独立上线，先计算简单战绩，再逐步扩展到复杂赛季账本。

------

## M8：履历与任免机制

### 6.33 目标

让所有关键岗位的任职历史、调整历史和阶段贡献可沉淀。

### 6.34 双轨履历

履历分为两类：

#### 完整履历

完整参与某项目全周期，或完整负责某一关键阶段。

#### 非完整履历

中途接任、中途调离、支援、替补、撤换、借调。

### 6.35 任命规则

- 军团负责发起作战军组建
- 主将与关键岗位由军团提出任命
- 任命生效后，关键岗位调整需经建管侧同意
- 所有变更必须写入履历
- 战绩按实际在岗期间累计

### 6.36 独立上线价值

履历与任免机制可以先与角色模块一起上线，不必等待完整结算引擎，就能沉淀“人”的长期资产。

------

## M9：地图与态势表达机制

### 6.37 目标

把复杂管理状态投射到地图与看板上，但不过度依赖复杂三维。

### 6.38 P1阶段表达重点

- 站班会点位
- 风险等级
- 作业活跃度
- 战区归属
- 热点事件
- 工单积压
- 健康度状态

### 6.39 P2阶段增强重点

- 变电站图层
- 输电线路图层
- 杆塔序列
- 工程骨架
- 支援流向

### 6.40 关键原则

当前MVP第二轮尚无实时推送、无线路轨迹、无后端，因此P1阶段不以重三维为重点，而以“管理态势表达增强”为重点。

### 6.41 独立上线价值

地图与态势机制可持续迭代，是所有其他机制的展示层，不必与底层能力完全同步推进。

------

## M10：规则引擎与申诉复核机制

### 6.42 目标

把系统从“看见问题”推进到“自动发现问题、自动触发管理动作”。

### 6.43 规则引擎

规则至少分三类：

#### 预警规则

- 进度偏差
- 长期无进展
- 高风险聚集
- 工单超期
- 频繁返工
- 关键岗位频繁变更

#### 结算规则

- 加分触发条件
- 扣分触发条件
- 冻结条件
- 年底归集条件

#### 履历规则

- 完整履历判定
- 阶段履历判定
- 调整事件权重
- 交接合法性检查

### 6.44 申诉复核

所有涉及扣分、责任归属、关键履历分类的动作，必须支持申诉。
申诉链建议为：

```
当事人申诉 → 督军府复核 → 现场督军意见 → 战区复核 → GM裁决
```

### 6.45 独立上线价值

即使先只上线最简单的预警规则和申诉冻结机制，也能显著提高系统的可信度和管理接受度。

## 八、分期上线策略

## 8.1 第一阶段：基础机制上线

上线模块：

- M1 赛季与目标
- M2 角色与席位
- M3 事件中心
- M9 地图与态势表达（轻量版）

目标：先让系统“有组织、有目标、有重点、有态势”。

## 8.2 第二阶段：闭环机制上线

上线模块：

- M4 军令与工单
- M5 现场日循环
- M6 健康度

目标：让系统“能下令、能跟踪、能闭环”。

## 8.3 第三阶段：评价机制上线

上线模块：

- M7 战绩与结算
- M8 履历与任免
- M10 规则引擎与申诉复核

目标：让系统“能评价、能复盘、能形成长期资产”。

## 8.4 第四阶段：增强联动上线

逐步接入：

- 实时推送
- 视频与感知设备
- 工程骨架图层
- 资源调度算法
- 智能辅助决策

该演进方向与公司顶设中关于现代智慧工地、无人机、多场景感知、现场管控微应用、工程智能指挥应用的规划是一致的。

------

## 九、边界约束

### 9.1 不做虚构资源系统

不引入脱离业务事实的金币、道具、虚拟兵种等娱乐化元素。

### 9.2 不做脱责式排行

排行必须建立在规则清晰、可申诉、可冻结的前提下。

### 9.3 不做一步到位的大一统平台

方案按机制包独立推进，允许先局部上线、再逐步集成。

### 9.4 不做脱离数据基础的重三维建设

在现阶段数据条件下，先做机制层和指挥层，不以重建模为先。

------

## 十、最终定义

基建online的最终定义是：

**一个以数字工单为底座、以四层组织为骨架、以事件与军令为驱动、以战绩与履历为沉淀、以地图与赛季为表达、并可按独立机制包灵活开发和独立上线的输变电工程调度指挥沙盘。**

它既继承了现有数字沙盘的态势展示能力，也承接了现代建设管理体系关于四大项目部、全过程精益化管理、分层调度指挥和数字工单闭环的管理逻辑，同时与建设专业数智化顶层设计中“业务线上流转 + 现场智能感知 + 远程集中监控”的方向保持一致。

这版已经按“相对独立机制”拆开，开发时可以按模块优先级灵活增删，不会因为某一块延迟而拖住整体上线。
