## 一、参考项目说明

参考的 `worldmonitor` 本质上是一个"实时态势看板"，核心特征包括：双地图引擎（3D globe + WebGL flat map）、多数据层叠加、实时数据流、统一态势界面；技术上用了 `globe.gl`、`deck.gl`、`MapLibre GL`、Vite/TypeScript。

但它的许可证是 **AGPL-3.0 非商业使用**，README 也明确写了商业使用 / SaaS / 重品牌需要商业授权，所以你可以参考它的架构思路和交互形态，但不要直接拿代码进正式商用项目。

**MVP 第一轮建议**: 参考 worldmonitor 的布局思路（地图为中心、左右面板、底部统计），但不要照搬其技术栈（不引入 globe.gl、deck.gl 等重型库）。

## 二、MVP 第一轮技术实现建议

### 2.1 技术栈选型

| 层级 | 选型 | 说明 |
|------|------|------|
| 地图引擎 | **MapLibre GL JS** | 开源、矢量渲染、轻量级 |
| 底图来源 | **天地图**（首选） | 需申请 Key，显示版权归属 |
| 前端框架 | React + TypeScript | 组合式 API，类型安全 |
| 构建工具 | **Vite** | 快速开发，热更新 |
| 样式方案 | **TailwindCSS** | 原子化样式，快速迭代 |
| 状态管理 | **Pinia** | 状态集中管理，调试友好 |
| 图表库 | **ECharts** | 统计图表，成熟稳定 |

### 2.2 数据源架构

**MVP 第一轮采用最简单的数据流**:

```
静态 JSON 文件
→ 前端读取解析
→ 浏览器内存存储
→ 组件响应式展示
```

- **数据源**: 本地静态 JSON（`/public/data/20260401.json`）
- **存储**: 浏览器内存（Pinia Store），无后端、无数据库、无持久化
- **计算**: 前端实时计算筛选和统计，无服务端聚合

### 2.3 前端架构设计

**功能模块划分**:

```
页面级
├── 地图区域（底图 + 边界 + 作业点标记）
├── 左侧面板（筛选条件）
├── 右侧面板（详情展示）
└── 底部面板（统计指标）
```

**数据分层**:

| 层级 | 数据内容 | 说明 |
|------|----------|------|
| **原始数据层** | 从 JSON 读取的原始记录 | 未经任何处理 |
| **清洗归一化层** | 坐标合法、人数归一化后的记录 | 用于全量展示和统计 |
| **当前筛选结果层** | 应用筛选条件后的子集 | 用于地图渲染和实时统计 |

**状态管理建议**:
- 数据层状态：原始数据、清洗后数据
- 交互层状态：当前筛选条件、选中标记、地图视图

### 2.4 数据清洗与归一化

**前端数据解析时执行**:

1. **坐标清洗**: 过滤掉经纬度为空、非法、超范围的记录（这些记录不上图、不统计）
2. **人数归一化**: 
   - 空/非数字/负数 → 半径值=0，统计值=0（丢弃）
   - 0-300 → 半径值=原值，统计值=原值
   - >300 → 半径值=300（截断），统计值=原值（保留）
3. **状态归类**: 脏值/空值 → "未知状态"
4. **风险归类**: 空值/非法值 → "未知风险"

清洗后的数据进入"清洗归一化层"，所有展示和统计均基于此。

### 2.5 关键实现注意点

**地图标记渲染**:
- 使用 MapLibre GL JS 的 `CircleLayer` 或 `SymbolLayer`
- 点大小通过 `circle-radius` 的表达式根据人数映射
- 点颜色通过 `circle-color` 的表达式根据风险等级映射

**筛选实现**:
- 筛选条件变更时，前端基于内存数据实时过滤
- 无需后端请求，响应时间 < 500ms

**统计计算**:
- 基于当前筛选结果实时计算
- 使用 JavaScript 数组方法（filter/reduce/map）即可满足性能需求

### 2.6 第一轮明确不引入的技术

以下技术虽在后续阶段可能有用，但MVP第一轮**明确不引入**:

| 技术 | 不引入理由 | 计划引入阶段 |
|------|------------|--------------|
| deck.gl | 数据量小（317条），无需海量点渲染 | 后续阶段（>10k点）|
| Three.js / globe.gl | 第一轮只做2D地图 | 暂不承诺 |
| WebSocket | 静态数据，无需实时推送 | MVP 第二轮 |
| 后端服务 | 静态JSON直接读取，无需API层 | 后续阶段 |
| 数据库 | 浏览器内存足够，无需持久化 | 后续阶段 |
| 数据接入网关 | 无多源接入需求 | 后续阶段 |

## 三、后续演进路线（MVP第二轮及以后）

### 3.1 MVP 第二轮扩展

**目标**: 时间轴与历史回溯

**新增技术需求**:
- 时间轴组件（基于 D3 或自定义实现）
- 数据切片逻辑（按小时分组）
- 播放/暂停控制（setInterval/requestAnimationFrame）

**数据**: 仍使用静态JSON，但需包含多时段数据

### 3.2 后续阶段完整架构

当系统需要接入实时数据、多源融合时，建议采用以下分层架构：

**第一层：数据接入层**
- 统一接入网关（REST/WebSocket）
- 断线重连、心跳、订阅恢复
- 限流与熔断

**第二层：数据标准化与融合层**
- 适配器模式（EJJAdapter、GridCommandAdapter）
- 统一领域模型（Project、Site、Task、RiskEvent）
- 数据清洗与转换

**第三层：时空数据服务层**
- 关系库（PostgreSQL/MySQL）：项目、标段、组织
- 时序库（TimescaleDB/ClickHouse）：告警、状态变化、轨迹
- 缓存（Redis）：当前在线状态、最新位置
- 瓦片服务：行政区划、工程分布、热力图

**第四层：实时态势编排层**
- 合并多源实时流
- 生成统一"态势事件"
- 按工程/区域/专业分类广播

**第五层：数字沙盘前端层**
- MapLibre GL：2D底图、行政区、点位
- deck.gl：海量点、热力、路径、聚合（大数据量时）
- WebSocket Store：统一管理实时状态

### 3.3 演进路径建议

```
MVP 第一轮（当前）: 静态 JSON + 前端单体
           ↓
MVP 第二轮: 时间轴 + 多时段静态数据
           ↓
MVP 第三轮: 接入第一个实时数据源（WebSocket）
           ↓
后续阶段:   多源融合 + 完整时空服务 + 实时态势编排
```

**建议**: 严格按照阶段推进，不要为"未来可能的需求"提前引入复杂度。MVP 第一轮的目标是"跑通展示闭环"，而非"搭建完整架构"。
