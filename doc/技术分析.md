# 技术路线分析（SQLite 修订版）

**版本**: M0-lite / M1 v1.0  
**日期**: 2026-04-16  
**范围**: M0-lite（数据底座收口） + M1（空间主数据与真实骨架地图）

**阶段说明**:
- pre-MVP：地图展示与时间轴历史回溯已完成
- 当前：M0-lite + M1
- 后续：M2 项目聚合与地图交互、M3 组织与赛季等

## 一、当前阶段目标

### 1.1 M0-lite 目标
- 建立后端工程底座
- 建立年度目标、杆塔、meetlist 三类数据导入链路
- 建立 `prjCode / singleProjectCode / 空间对象` 映射
- 让现有前端地图页可从 API 取数

### 1.2 M1 目标
- 建立变电站、线路、杆塔、作业点主数据
- 基于杆塔生成线路几何链
- 在现有地图上加载站/线/塔/作业点四层

### 1.3 当前阶段不是
- 微服务拆分
- PostgreSQL / PostGIS
- WebSocket 实时推送
- 复杂规则引擎
- 事件工单与战绩履历
- 3D 地图或特效系统

## 二、技术栈选型（修订版）

### 2.1 前端

| 层级 | 选型 | 说明 |
|------|------|------|
| 地图引擎 | **MapLibre GL JS** | 继续复用现有地图底座 |
| 前端框架 | **React + TypeScript** | 与现有实现保持一致 |
| 构建工具 | **Vite** | 开发快，适合迭代 |
| 样式方案 | **TailwindCSS** | 快速搭建中后台布局 |
| 状态管理 | **Zustand** | 轻量状态管理 |
| 图表库 | **ECharts** | 保留统计能力 |

### 2.2 后端

| 层级 | 选型 | 说明 |
|------|------|------|
| 语言 | **Python 3.11+** | 学习成本低，开发效率高 |
| Web 框架 | **FastAPI** | 适合轻量后端与 API 层 |
| 数据建模 | **SQLModel** | 统一 ORM 与 schema 风格 |
| 数据库 | **SQLite** | 轻量、零部署、适合当前阶段 |
| 迁移工具 | **Alembic** | 保证版本演进 |
| 启动服务 | **Uvicorn** | 本地开发简洁 |

### 2.3 为什么改用 SQLite

当前阶段更适合 SQLite，而不是 PostgreSQL，原因如下：
- 当前仍是单机/本地开发为主
- M0-lite / M1 的核心是数据导入、映射、骨架生成，不是并发吞吐
- SQLite 零部署、上手快，适合 vibe coding 节奏
- 当前阶段无需 PostGIS、复杂事务并发、跨服务访问
- 后续如进入更大规模阶段，再迁移 PostgreSQL 即可

## 三、总体架构设计

### 3.1 架构原则

当前阶段采用：
**前后端分离 + 模块化单体 + 轻量数据库（SQLite）**

不直接拆微服务，不提前引入复杂中间件。

### 3.2 架构图

```text
Browser
  ↓
React + Vite + MapLibre
  ↓ HTTP
FastAPI
  ├── Import Module
  ├── Mapping Module
  ├── Spatial Module
  ├── Timeline Snapshot Module
  └── API Compatibility Layer
  ↓
SQLite
```

### 3.3 模块划分

#### M0-lite 模块
- Import Module：导入与覆盖
- Mapping Module：标识映射
- Snapshot Module：meetlist 小时快照与归档
- Compatibility Module：前端兼容 API

#### M1 模块
- Station Module：变电站主数据
- Tower Module：杆塔主数据
- Line Module：线路主数据与几何链
- WorkPoint Module：作业点与空间对象挂接
- Map Projection Module：前端地图数据投影

## 四、数据源架构

### 4.1 当前阶段数据源

| 数据源 | 来源 | 用途 |
|--------|------|------|
| 年度目标数据 | 文件导入 | 项目/单项工程年度目标 |
| 杆塔数据 | 文件导入 | 线路空间骨架 |
| meetlist 数据 | 文件导入 | 作业快照与时间轴变化 |

### 4.2 数据流

```text
源文件（JSON / CSV / 结构化文件）
→ 导入解析
→ 原始表保存
→ 业务表全覆盖更新 / 快照写入
→ API 输出
→ 现有前端页面消费
```

### 4.3 数据存储策略

#### 原始表
保存源文件原始内容与导入批次：
- `raw_year_progress`
- `raw_tower`
- `raw_meeting_snapshot`

#### 业务表
保存当前生效视图：
- `year_progress_current`
- `tower_current`
- `meeting_current`
- `station_current`
- `line_current`

#### 快照表
保存短期时间快照：
- `meeting_hourly_snapshot`
- `meeting_daily_archive`

## 五、SQLite 设计原则

### 5.1 使用原则
- 当前阶段默认单文件 SQLite
- 开发环境与测试环境共用一套 schema
- 不提前为高并发做重优化
- 不做复杂锁设计

### 5.2 适配原则
- 以结构清晰、易迁移为优先
- 避免使用过于依赖 PostgreSQL 的特性
- 几何链先用 JSON 文本或普通字段存储，不依赖 PostGIS

### 5.3 几何数据存储建议
在 SQLite 中：
- 站点：经纬度字段分列存储
- 杆塔：经纬度字段分列存储
- 线路：几何链使用 GeoJSON 字符串或 JSON 文本存储

当前阶段不引入 GIS 扩展，不引入复杂空间索引。

## 六、M0-lite 技术实现

### 6.1 目标
建立最小后端与数据底座，不新增业务页面。

### 6.2 核心任务

#### 6.2.1 后端初始化
- FastAPI 工程初始化
- SQLite 连接与会话管理
- SQLModel 模型定义
- Alembic 迁移框架

#### 6.2.2 导入能力
- 年度目标导入
- 杆塔导入
- meetlist 导入
- 导入日志记录
- 原始表写入
- 业务表覆盖更新

#### 6.2.3 覆盖规则
- `tower` 与 `yearProgressFormation` 同主键全覆盖业务表全部字段
- meetlist 以 `id` 为主键维护最新快照
- meetlist 24 小时内保留小时快照，其余按天归档

#### 6.2.4 API 兼容层
建议先提供最小接口：
- `GET /health`
- `GET /api/bootstrap`
- `GET /api/map/summary`
- `GET /api/meeting/latest`

### 6.3 当前阶段不做
- 复杂业务 API
- 后台管理页面
- 规则引擎
- 事件工单服务
- 复杂权限系统

## 七、M1 技术实现

### 7.1 目标
在现有地图底座上建立真实工程骨架。

### 7.2 核心任务

#### 7.2.1 站点主数据
- 建立站点主表
- 复用已有稳定站点主键
- 支持接入关系对象

#### 7.2.2 杆塔主数据
- 建立杆塔主表
- 存储塔号、序号、上游塔号、经纬度
- 保留异常数据记录

#### 7.2.3 线路主数据
- 建立线路主表
- 支持一个 `singleProjectCode` 对应多条几何链
- 线路几何以 JSON 文本形式存入 SQLite

#### 7.2.4 作业点挂接
- 建立作业点与站/线/塔/单项工程的挂接关系
- 输出最新值与 `+n/-n`

#### 7.2.5 线路生成服务
输入：
- `singleProjectCode`
- `towerSequenceNo`
- `upstreamTowerNo`
- 坐标字段

输出：
- 一条或多条 GeoJSON 折线
- 异常链清单

### 7.3 当前阶段图层
- `StationLayer`
- `LineLayer`
- `TowerLayer`
- `WorkPointLayer`

这些图层在前端继续由 MapLibre 加载，不另起重型地图引擎。

## 十、风险与控制

### 10.1 SQLite 风险

| 风险 | 说明 | 控制策略 |
|------|------|----------|
| 并发能力弱 | 不适合高并发写入 | 当前阶段本地开发可接受 |
| 空间能力弱 | 无 PostGIS 能力 | 线路几何先用 JSON 文本存储 |
| 后续迁移成本 | 未来可能需要迁移 PostgreSQL | 从一开始保持清晰模型和迁移边界 |

### 10.2 数据质量风险

| 风险 | 控制策略 |
|------|----------|
| 杆塔链不稳定 | 输出异常链清单，不阻断主流程 |
| 站点未匹配 | 允许 `unmatched` 容错 |
| meetlist 变化过多 | 用 `id` 做主键，变化值只做展示 |

## 十二、当前阶段结论

M0-lite / M1 当前最适合采用：
- **React + TypeScript + MapLibre**
- **FastAPI + SQLModel + SQLite**

这套技术组合的重点是：
- 容易启动
- 易于 vibe coding
- 足够支撑数据导入、映射和真实骨架地图
- 不会因为过重技术选型拖慢当前阶段
