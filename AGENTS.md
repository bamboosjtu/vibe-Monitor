# agent.md

## 1. 角色定位

你是本项目的实现型开发 Agent，负责基于已冻结文档完成 MVP 第一轮开发。

你的首要目标不是“做出一个炫酷的大屏”，而是：

1. 严格落实《需求分析.md》中的业务边界与验收口径
2. 严格参考《技术分析.md》中的阶段性技术路线
3. 以最小复杂度完成“当前作业上图闭环”
4. 为后续阶段预留合理扩展空间，但**不提前实现后续能力**

---

## 2. 项目背景

本项目为“输变电工程数字沙盘系统”MVP 第一轮。

当前阶段目标：

* 基于静态样例数据完成“当前作业上图闭环”
* 地图中心展示湖南省范围
* 渲染站班会点位
* 支持筛选、详情、统计联动

当前阶段不是实时态势系统，不是 3D 数字孪生系统，也不是多源融合平台。

---

## 3. 开发范围（只允许做这些）

## 5. 技术方向约束

### 5.1 前端技术方向

优先采用以下技术栈：

* React
* TypeScript
* Vite
* TailwindCSS
* Zustand 或 Redux Toolkit（二选一，优先简单方案）
* MapLibre GL JS
* ECharts

说明：

* 原冻结需求文档中曾写 Vue 3 + Pinia，但当前开发阶段统一改为 **React 技术栈优先**。
* 若某一实现细节因 React 方案更优而需要调整实现方式，可以调整，但不得改变需求口径与页面功能边界。

### 5.2 后端技术方向

MVP 第一轮默认**不建设后端服务**。

若后续开发过程中必须增加本地开发辅助服务或脚本，优先采用：

* Python 3.11+
* FastAPI（仅当确有必要）
* Pydantic
* Uvicorn

注意：

* 本轮不主动建设业务后端
* 不主动引入数据库
* 不主动实现 API 网关
* Python 仅作为未来扩展的优先后端技术方向与辅助脚本方案

## 6. 实现优先级原则

开发时必须遵循以下优先级：

1. **需求正确性优先**
2. **数据口径一致性优先**
3. **联动闭环优先**
4. **异常状态健壮性优先**
5. **视觉美化最后处理**

禁止为了“更好看”破坏需求边界、口径一致性和实现节奏。

---

## 7. 数据处理原则

你必须严格遵守冻结需求中的数据规则：

### 7.1 坐标处理

* 经纬度为空、非法、超范围、为 0 的记录：过滤
* 被过滤记录不上图、不参与统计

### 7.2 人数处理

* 空值 / 非数字 / 负数：视为 0
* 大于 300：

  * 半径映射值按 300 截断
  * 统计与详情展示保留原值

### 7.3 状态处理

* 脏值 / 空值：归类为“未知状态”
* 仅在“全部”状态下展示

### 7.4 风险处理

* 空值 / 非法值：归类为“未知风险”
* 未知风险需采用灰色样式
* 风险分布中单列统计

### 7.5 同坐标多记录

* 第一轮允许重叠
* 点击仅展示最上层一条记录
* 仅在命中重叠坐标时显示提示文案

---

## 8. 统计口径原则

所有统计必须严格遵守冻结需求文档中的定义：

* 当日作业点总数：合法坐标记录数
* 当前作业总人数：合法坐标记录的归一化统计值之和
* 风险分布：按记录数统计
* 市州分布 Top5：按记录数统计，不含“其他”

必须保证：

`地图标记数量 == 当日作业点总数`

---

## 9. 页面与交互原则

必须实现以下页面结构：

* 左：筛选面板
* 中：地图区域
* 右：详情面板
* 底：统计面板

必须覆盖以下状态：

* loading
* 数据加载失败
* 空数据
* 筛选无结果

1. 项目初始化与基础框架

## 11. 代码质量要求

### 11.1 必须做到

* TypeScript 类型明确
* 避免 any
* 数据处理逻辑与 UI 展示逻辑分离
* 统计逻辑集中管理
* 样式与业务逻辑尽量分离
* 组件职责单一

### 11.2 不建议出现

* 巨型单文件组件
* 在多个组件内重复实现同一统计逻辑
* 组件直接处理原始 JSON 脏数据
* 把地图实例、筛选逻辑、统计逻辑全部塞进 App 组件
